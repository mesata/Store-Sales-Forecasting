# Walmart---Store-Sales-Forecasting
ეს არის მანქანური სწავლების (Machine Learning) პროექტი, რომლის მიზანია **Walmart**-ის მაღაზიების და დეპარტამენტების ყოველკვირეული გაყიდვების პროგნოზირება ისტორიული მონაცემების, სეზონურობისა და სხვადასხვა ეკონომიკური ფაქტორების (CPI, Unemployment და ა.შ.) საფუძველზე.

ამ მოდელის გაწვრთნისას თავიდან დავიწყე base feature engineering-ით (`WalmartTransformer` კლასი), რომელიც მოიცავდა მარტივი ფიჩერების შექმნას: `Year`, `Month`, `Day`. ეს გავაკეთე, რადგან ერთი დროის ფიჩერი დამეშალა და მოდელისთვის მეტი საშუალება მიმეცა დროის აღქმისთვის. ამ feature engineering-ზე დაყრდნობით გავუშვი შემდეგი run-ები:

### ეტაპი 1: ჰიპერპარამეტრების ოპტიმიზაცია და ბაზისური ფიჩერები

| მოდელის სახელი (MLflow Run) | მიდგომა / ძირითადი პარამეტრები | Train WMAE | Validation WMAE | 
| :--- | :--- | :--- | :--- |
| **XGBoost_Baseline** | საწყისი ფიჩერები, `n_estimators=100`, `max_depth=6`, `lr=0.1` | $4,052.25 | $3,977.81 |
| **XGBoost_Hyperparameter_Tuning** | `n_estimators=300`, `max_depth=7`, `lr=0.03`, `subsample=0.8` | $3,748.08 | $3,744.20 | 
| **XGBoost_Advanced_Tuned** | `n_estimators=400`, `max_depth=8`, `lr=0.05`, `gamma=0.2`, `min_child_weight=3` | $2,408.34 | $2,832.63 |
| **XGBoost_New_Feature_Engineering** | **Advanced ჰიპერპარამეტრები + კალენდარული ფიჩერები (IsHoliday, Is_December, Is_November, Week)** | $2,301.18 | **$2,697.85** | 

მხოლოდ ჰიპერპარამეტრების სწორმა რეგულარიზაციამ (`gamma`, `min_child_weight`) და ხეების რაოდენობის ზრდამ ვალიდაციის შეცდომა $3,977-დან $2,832-მდე ჩამოაგდო.

### ეტაპი 2: სეზონური Feature Engineering (Advanced + FE)

შედეგი მაინც არ მეჩვენა საკმარისი და ვცადე შემეცვალა Feature Engineering მიდგომა. გავაკეთე ახალი `WalmartDataTransformer_updatedFeatureEngineering` კლასი, დავამატე ახალი `Is_December` და `Is_November` ფიჩერები, რადგან როგორც დატაზე დაკვირვებისას ვნახეთ, ამ პერიოდში გაყიდვები ყველაზე მეტად იზრდებოდა და მინდოდა მოდელს ამ კანონზომიერებისთვისაც მიექცია ყურადღება. 

| მოდელის ტიპი | პარამეტრები | Train MAE | Validation MAE |
| :--- | :--- | :--- | :--- | 
| **XGBoost (Advanced + FE)**| **Advanced + Holiday & Monthly Features** | $2,223.33 | **$2,697.30** |

**Feature Engineering-ის ეფექტი:** გაყიდვების დეტალური ანალიზის საფუძველზე შექმნილმა სპეციალურმა სეზონურმა ინდიკატორებმა (`Is_December`, `Is_November`) მოდელს მისცა მნიშვნელოვანი ინფორმაცია პიკური პერიოდების დასაჭერად. შედეგად, მოდელმა მიაღწია **WMAE: $2,697.85**.

### Feature Importance

საუკეთესო მოდელიდან (`XGBoost Advanced + FE`) ამოვიღე ფიჩერების მნიშვნელოვნების კოეფიციენტები. გრაფიკი ნათლად აჩვენებს, თუ რა ფაქტორებზე დაყრდნობით პროგნოზირებს მოდელი Walmart-ის გაყიდვებს:

<img width="890" height="513" alt="image" src="https://github.com/user-attachments/assets/7ea3ec9a-aca1-434f-ac0a-aa97943ba61f" />

* **Dept (დეპარტამენტი):** პირველ ადგილზეა. ეს ლოგიკურია, მაგალითად, ელექტრონიკის ან სათამაშოების დეპარტამენტს სულ სხვა გაყიდვები აქვს, ვიდრე პროდუქტებისას. მოდელმა ეს ყველაზე მნიშვნელოვან ფაქტორად აღიქვა.
* **Type და Size:** დიდი მაღაზიები ბევრად მეტს ყიდიან, ვიდრე პატარა, უბნის Walmart Express-ები.
* **სეზონური ფიჩერები (`Is_December`, `Is_November`, `Week`):** დიდი როლი ითამაშეს მოდელის სიზუსტის გაზრდაში, რადგან მათ დაიჭირეს ნოემბრის (Thanksgiving) და დეკემბრის (Christmas) პერიოდებში არსებული აქტივობა მაღაზიებში.
* **`CPI` და `Unemployment`:** ნაკლებად ახდენენ მოკლევადიან მერყეობებს გაყიდვებში, რაც Walmart-ის, როგორც სტაბილურ მდგრადობაზე მიუთითებს.

### ეტაპი 3: საბოლოო მოდელი — დროითი რიგების ფიჩერები (Lag Features & Rolling Mean)

დროითი მწკრივის სპეციფიკიდან გამომდინარე, პროგნოზირების ხარისხის კიდევ უფრო გასაუმჯობესებლად შემოვიტანე **Lag ტრანსფორმატორი** (`WalmartLagTransformer`), რომელმაც თითოეული მაღაზიისა და დეპარტამენტების ჭრილში დააგენერირა წინა კვირების გაყიდვების მაჩვენებლები (`Lag_1`, `Lag_2`, `Lag_4`) და მოძრავი საშუალო (`Rolling_Mean_4`).

ეს პროცესი სრულად გაერთიანდა ერთიან **All-In-One Pipeline**-ში, რამაც თავიდან აგვაცილა მონაცემთა გაჟონვა (Data Leakage) და მოდელი გახადა მარტივად გამოსაყენებელი ახალ სატესტო მონაცემებზე.

ამ ეტაპზე გაეშვა ორი ძირითადი ექსპერიმენტი Overfitting-ის კონტროლისთვის:

| მოდელის სახელი (MLflow Run) | მიდგომა / პარამეტრები | Train WMAE | Validation WMAE | სხვაობა (Variance) |
| :--- | :--- | :--- | :--- | :--- |
| **XGBoost_with_Lags** | საწყისი ოპტიმიზებული პარამეტრები | $1,285.89 | $1,491.65 | ~$205.76 (Overfitting) |
| **XGBoost_with_Lags_diff_parameters** | **ძლიერი რეგულარიზაცია (`gamma=0.1`, `min_child_weight=3`)** | $1,517.06 | **$1,499.40** | **~$17.66 (სტაბილური)** |

#### დასკვნა და ეფექტი:
მიუხედავად იმისა, რომ პირველ ექსპერიმენტს ვალიდაციაზე უმცირესი უპირატესობა აქვს ($1,491 vs $1,499), მასში მკაფიოდ ჩანს გადატრენინგების (Overfitting) ნიშნები. რეგულარიზაციის პარამეტრების ჩართვით (`XGBoost_with_Lags_diff_parameters`), მოდელმა შეინარჩუნა მაღალი სიზუსტე და ამავდროულად მიაღწია თითქმის იდეალურ ბალანსს Train და Validation შეცდომებს შორის (სხვაობა მხოლოდ $17). 

ეს ვერსია საბოლოოდ შეირჩა საუკეთესო პროდუქციულ მოდელად, რადგან ის ბევრად უფრო მდგრადია ახალი, უცნობი მონაცემების მიმართ.
